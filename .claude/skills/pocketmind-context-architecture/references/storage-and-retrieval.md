# 存储与检索设计

## 总体原则
PocketMind 适合“业务数据在数据库、上下文内容走混合模型”的方式，而不是所有东西都文件化。

## Note / Chat 主数据
保留在数据库：
- 便于客户端高频读取
- 便于同步、分页、过滤、排序
- 保持现有移动端体验稳定

## Resource / Memory / Skill
建议采用“双层存储”思路：

- 数据库存索引、状态、关系、引用
- 内容正文按资源类型选择 DB 文本或文件/对象存储

## 为什么不把分享帖子正文全部落文件
因为：
- 客户端会直接读正文
- 需要同步与搜索
- 需要高频读取和投影展示

因此：
- `WEB_CLIP` 正文主存可以是数据库文本
- L0/L1 可以是数据库投影
- 文件层不是必须

## L0 / L1 / L2 的 PocketMind 解释
- L2：原始正文 / 原始消息 / 原始提取文本
- L1：概览 / 中等摘要 / 场景理解
- L0：极短摘要 / 主题标签 / 快速判断信息

## 检索层建议
后续检索建议统一通过 `context_catalog` 之类的索引表进入，再决定是否读取正文。

不要让 AI 查询直接扫描业务表，更不要扫描文件目录作为第一步。
